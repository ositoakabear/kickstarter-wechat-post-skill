# 写作风格与结构模板

通用原则：不要原样照抄源文案；按平台风格精简改写。两个平台内容保持事实一致，**改一处同步另一处**。

## 通用写作细节

- 专业英文术语首次出现时写成 `English（中文解释）`，例如 `Mesh（网状网络）`、`MIPS（多方向冲击保护系统）`、`Open-Ear Audio（开放式音频）`。同一篇文章后文不再重复括注，除非隔了很远且读者可能遗忘。
- 封面图、标题图和正文图里的可见文案也遵守术语规则：能用中文就用中文；必须保留英文时，只保留少量关键词，避免整张图像产品参数表。
- 创始团队介绍不要只写“某某团队很厉害”。适合单独成节，至少交代品牌/公司、创始人或核心团队、背景来源、产品动机、过往作品/众筹履历、履约记录和媒体/节目背书；必须放一张官方团队/创始人图，并在图注写清来源。找不到团队合照时，用官方故事页、采访或媒体报道中的创始人早期合影替代；仍找不到时，明确说明未找到可信团队图，不要用无关氛围图冒充。
- 观察员点评要更像一个人的判断，不要像 AI 总结。先给观点，再给理由，最后落到产品哲学或用户行为变化。可以更思辨：讨论产品为什么成立、它触碰了什么真实动作、哪些功能是“用户可以不用，但关键时刻必须得有”。

## 过往成稿沉淀

优先使用 `PART + REVIEW + LINKS` 的公众号骨架。xLean / Lumos 这类信息完整的文章按这个骨架走；Birdie 这类短稿可以保留强钩子，但不要回退成只有卖点罗列、缺少团队/风险/资料导航的轻内容。

- **开场**：第一屏先给产品反差、奖项、众筹成绩或一句清晰判断，不要先铺大段背景。标题统一用 `本周好物推荐：<产品名/钩子>`。
- **PART 01｜创始团队/故事由来**：团队信息足够时尽量前置，讲品牌、创始人、产品动机、过往产品/履约记录、媒体或奖项背书，并配团队/创始人图。不要把团队背景藏在文末。
- **PART 02｜项目概览**：用固定字段帮助读者快速扫读：`项目名称`、`上架时间`、`众筹表现`、`参考价格`、`预计交付`、`核心功能`。核心功能用 `●` 自定义段落，不用 `<ul>/<li>`。
- **PART 03｜真实痛点**：用一句“它真正解决的是……”收束，不只复述功能。痛点要落到具体场景、具体动作、具体不便。
- **PART 04｜功能与体验**：围绕 2-4 个关键场景展开，每个场景配图。状态对比、前后变化、开合/变形等内容要写清左右或先后，避免图片摆反。
- **PART 05｜竞品对比**：不用横向大表，改成竖向维度卡片。每张卡只比较一个维度，例如安装方式、交互方式、维护成本、适用人群、价格区间；本品优势用 `🏆` 或醒目底色突出。
- **PART 06｜风险、目标受众与市场判断**：风险要具体到这个项目，例如交付时间、耗材/订阅、售后、规格兼容、运输安装、App 依赖、儿童/宠物/户外等特殊场景限制。目标受众单独写，不要只说“适合所有人”。
- **REVIEW｜观察员点评**：不是复述全文，而是给一个有判断的观点：为什么这个产品成立，它和同类产品的关键差异是什么，它可能代表了什么消费趋势。
- **LINKS｜资料导航**：固定放 Kickstarter、官网、众筹跟踪页、媒体报道、品牌故事/创始人页。移动端链接可能不可点击，必要时直接写纯文本网址并加 `word-break:break-all`。

## 小红书 markdown

风格：第一人称、口语化、轻 emoji、短句、有「被种草」感。控制在手机一屏多滑几下能读完。

结构模板：

```markdown
# 本周好物推荐：<钩子标题，制造好奇/反差>

<开场一句话点出产品 + 最大亮点/获奖背书>

(配图 1: <主图说明，如 实物挂墙图>)

🐦 **<亮点小标题>：**
<2-4 行解释，口语化>

(配图 2: <动态/GIF 说明>)

✨ **<对比/状态小标题>：**
- **【常态】** ...
- **【死态】** ...

(配图 3: <成对对比图说明>)

✨ **核心升级：**
1. **<点>**：...
2. **<点>**：...

(配图 4: <App/细节图说明>)

📊 **货比三家：**
- **<本品>**：... | ... | ...
- **<竞品A>**：...
- **<竞品B>**：...

🐻 **"<栏目名>"观察员<昵称>点评：**
> "<一句有观点的总结>"

⚠️ **众筹小贴士**：Kickstarter 项目存在交付延期与售后不确定性，理性支持。

#垂直标签1 #垂直标签2 #品类 #人群 #本周好物推荐
```

要点：
- 用 `(配图 N: 说明)` 占位，正文里讲清每张图位置与内容。
- 成对状态（常态/死态、前/后）要写明左右或先后，避免后续图摆反。
- 标签：3-9 个，含品类 + 人群 + 固定栏目标签。

## 公众号 HTML

风格：比小红书更完整、信息更全，但仍要精简有节奏。分小节，配图穿插。排版用内联样式（微信不支持外部 CSS / class 不稳）。**渲染坑详见 wechat-pitfalls.md**。

建议结构（每节一个 `<section>`）：
1. 开场 + 获奖/数据背书（封面图在草稿封面，不在正文顶部重复）
2. `PART 01` 创始团队/品牌背景（配团队或创始人图，讲清产品动机、履约记录、过往众筹/媒体背书）
3. `PART 02` 项目概览（项目名、上架时间、众筹表现、参考价格、预计交付、核心功能）
4. `PART 03` 真实痛点/创意由来（明确“真正解决的是……”）
5. `PART 04` 功能与体验：成对状态图用 `<table>` 并排，文字在图下；Pro/升级点用自定义 `●` span 列表
6. `PART 05` **竞品对比：竖向卡片**，每个维度一张卡，本品高亮 `🏆`
7. `PART 06` 风险、目标受众与市场判断（风险要产品专属，受众要具体）
8. `REVIEW` 「观察员点评」引用块（署名按用户约定）
9. `LINKS` 官方资源导航（链接）+ 众筹风险提示

标题：统一前缀便于系列化，如 `本周好物推荐：<标题>`。

正文配图：图片先下载或生成到 `.../Kickstarter/<产品名>/` 产品素材文件夹，HTML 里优先用这些本地图片路径；脚本会自动上传到微信 CDN 并替换（见 SKILL.md 第 6 步）。

观察员点评参考语气：

> Lumos 虽本质上是将一堆功能接入了一个头盔，但比起在华强北已经卷烂的各种 N 合一低价白牌电子产品，它明显很认真地观察并处理了骑行中“小但要命”的沟通问题。这也是我十分信奉的 ToC 产品哲学，产品的上限取决于你有没有那些“用户可以不用，但你必须得有”的功能，更何况它与安全息息相关。
>
> 很多时候，安全不通过 spec 体现，而在于队友那句“前面有坑”“我掉队了”“准备左转”能不能及时传出去。Lumos 把头盔从被动防护做成会发声、会示意、会连队友的节点，这个方向很像骑行装备的下一步：不只是更电子化，而是更像一个可靠的同伴。
